AIRS Field Class

BLE로 노드에 Wi-Fi 연결하기

팀원이 같은 그림을 보고 이해할 수 있도록, Android 앱이 BLE로 AIRS 노드에 SSID/PW를 전달하고 노드가 직접 Wi-Fi에 연결되는 실제 검증 절차를 단계별로 정리했습니다.

1. 오늘 검증의 성공 기준

앱 기준

노드 검색

앱 목록에 AIRS-SETUP-XXXX 노드가 나타나야 합니다.

BLE 기준

GATT 연결

앱이 AIRS Service UUID를 찾고 Wi-Fi Config Characteristic에 write해야 합니다.

노드 기준

Wi-Fi 연결

노드가 전달받은 SSID/PW로 Wi-Fi에 연결되면 provisioning 성공입니다.

2. 현재 검증 상태

제품 구현 기준은 Android 휴대폰입니다. 현재 Android 휴대폰이 없을 때 Samsung 태블릿을 임시 테스트 장비로 사용할 수 있지만, 문서와 앱 흐름은 휴대폰 기준으로 유지합니다. 실제 AIRS 노드가 아직 없으므로 노드 advertising, GATT 연결, Wi-Fi 연결, MQTT publish는 노드 확보 뒤 별도로 성공 확정해야 합니다.

성공 확정

노드 없이 검증한 것

  • Kotlin + Jetpack Compose Android 프로젝트 생성
  • MVVM + Repository 기반 Wi-Fi provisioning 흐름 구성
  • Android 휴대폰용 실제 Android BLE 권한 요청 화면 구현
  • Native Android BLE API 기반 scan, GATT connect, write, notify repository 구현
  • mock 노드 검색, 노드 선택, 연결 중, Wi-Fi 입력, 전송 중, 성공/실패 화면 전환
  • SSID/PW JSON payload 생성 및 비밀번호 masking preview
  • mock 성공 흐름: received -> wifi_connecting -> wifi_connected
  • mock 실패 흐름: SSID에 fail 포함 시 wifi_failed
  • 단위 테스트와 debug build 검증
아직 미확정

실물로 확인해야 하는 것

  • Android 휴대폰에서 앱 설치, 권한 승인, Bluetooth ON 상태 확인
  • Android 휴대폰에서 실제 AIRS 노드 BLE scan 결과가 표시되는지
  • AIRS 노드 advertising 이름과 Service UUID 일치 여부
  • GATT 연결, service discovery, characteristic property 확인
  • Wi-Fi Config Characteristic에 실제 JSON write 성공 여부
  • Status Characteristic notify로 wifi_connected/wifi_failed 수신 여부
  • 노드가 전달받은 SSID/PW로 실제 2.4GHz Wi-Fi에 연결되는지
  • Wi-Fi 연결 이후 MQTT broker/backend/InfluxDB로 sensor data가 들어오는지

3. 성공 흐름: 실제 코드 추적

아래 표는 AIRS 앱이 노드 BLE를 발견하고, GATT로 연결한 뒤, Wi-Fi 정보를 write하고, 노드가 wifi_connected를 notify했을 때 어떤 코드가 어떤 UI를 보여주는지 따라가는 지도입니다. Samsung 태블릿으로 임시 테스트해도 앱 내부 흐름은 동일합니다.

MainActivity -> ProvisioningRoute -> ProvisioningScreen -> ProvisioningViewModel -> AndroidBleProvisioningRepository -> BluetoothLeScanner/BluetoothGatt -> AIRS Node -> status notify -> ProvisioningViewModel -> ResultContent
흐름 실제 코드 위치 사용되는 함수/변수 보이는 UI / 디버깅 기준
앱 시작과 권한 확인 mobile/android/app/src/main/java/com/airs/android/MainActivity.kt blePermissionsGranted, blePermissionLauncher, BleRuntimePermissions.hasAll() 권한이 없으면 BlePermissionScreen이 보입니다. Bluetooth 권한 허용 후 provisioning 화면으로 넘어가야 합니다.
권한 화면 provisioning/ui/BlePermissionScreen.kt BlePermissionScreen(onRequestPermissions) Bluetooth 권한 필요 카드와 Bluetooth 권한 허용 버튼이 보입니다. 버튼은 MainActivity.requestBlePermissions()로 이어집니다.
첫 화면 표시 MainActivity.kt, provisioning/ui/ProvisioningScreen.kt ProvisioningRoute(viewModel), ProvisioningScreen(), uiState.step uiState.step == Start이면 StartContent가 보이고, 사용자는 AIRS 노드 검색 시작 버튼을 누릅니다.
검색 버튼 클릭 ProvisioningScreen.kt, ProvisioningViewModel.kt StartContent(onStartScan), viewModel.startScan(), uiState = ProvisioningUiState(step = Scanning) 화면은 ScanningContent로 바뀌고 BLE 스캔 중 progress bar가 보입니다.
실제 BLE scan 시작 provisioning/data/AndroidBleProvisioningRepository.kt scanForNodes(), BluetoothLeScanner.startScan(), ScanCallback.onScanResult() Android BLE API가 주변 advertising을 받습니다. 노드가 없으면 SCAN_TIMEOUT_MILLIS 뒤 실패 메시지가 나오는 것이 정상입니다.
AIRS 노드 필터링 AndroidBleProvisioningRepository.kt, AirsBleConstants.kt ScanResult.toAirsNode(), NODE_NAME_PREFIX, PROVISIONING_SERVICE_UUID, AirsBleNode 노드 이름이 AIRS-SETUP-로 시작하거나 Service UUID가 맞으면 AirsBleNode로 변환됩니다.
노드 목록 UI 표시 ProvisioningViewModel.kt, ProvisioningScreen.kt repository.scanForNodes().collect { nodes }, uiState.nodes, NodeListContent, NodeCard uiState.step == NodeList가 되고 발견된 AIRS-SETUP-XXXX 카드가 보입니다.
노드 선택과 GATT 연결 ProvisioningScreen.kt, ProvisioningViewModel.kt, AndroidBleProvisioningRepository.kt NodeCard(onNodeSelected), selectNode(node), connect(node), device.connectGatt() 화면은 ConnectingContent로 바뀝니다. 연결 실패 시 실패 화면과 오류 메시지가 보입니다.
Service/Characteristic 확인 AndroidBleProvisioningRepository.kt, AirsBleConstants.kt onConnectionStateChange(), discoverServices(), onServicesDiscovered(), cacheAirsCharacteristics() AIRS Service, Wi-Fi Config Characteristic, Provision Status Characteristic을 찾으면 WifiInput 화면으로 이동합니다.
Wi-Fi 정보 입력 ProvisioningScreen.kt, ProvisioningViewModel.kt WifiNetworkPicker, loadWifiNetworks(), selectWifiNetwork(), uiState.ssid, uiState.password 현재/주변 Wi-Fi SSID 후보를 보여주고, 사용자가 선택하면 SSID가 채워집니다. Android는 저장된 Wi-Fi 비밀번호를 앱에 제공하지 않으므로 비밀번호는 직접 입력해야 합니다.
Wi-Fi payload 전송 ProvisioningViewModel.kt, WifiConfigPayload.kt, AndroidBleProvisioningRepository.kt sendWifiConfig(), WifiCredentials, WifiConfigPayload.build(), provisionWifi(), writeWifiPayload() 화면은 ProvisioningProgressContent로 바뀝니다. 앱은 {"ssid":"...","password":"..."} JSON을 Wi-Fi Config characteristic에 write합니다.
Status notify 수신 AndroidBleProvisioningRepository.kt, ProvisioningViewModel.kt enableStatusNotifications(), onCharacteristicChanged(), handleStatusBytes(), toProvisioningStatus(), collect { status } 노드가 received, wifi_connecting, wifi_connected를 보내면 StatusList에 순서대로 쌓입니다.
성공/실패 화면 결정 ProvisioningViewModel.kt, ProvisioningScreen.kt when (status.type), ProvisioningStatusType.WifiConnected, ProvisioningStep.Success, ResultContent wifi_connected를 받으면 Wi-Fi 설정 완료 화면이 보입니다. wifi_failed를 받으면 실패 화면이 보입니다.
디버깅할 때는 화면이 멈춘 위치를 먼저 보고, 그 위치의 uiState.step과 위 표의 해당 함수부터 확인하면 됩니다. 예를 들어 노드 목록이 안 뜨면 scanForNodes(), onScanResult(), toAirsNode(), uiState.nodes 순서로 보면 됩니다.

5. Samsung 태블릿 임시 실행 Step

AIRS 앱의 제품 기준은 Android 휴대폰입니다. 다만 노드 확보 전까지 Samsung 태블릿을 임시 실기기 장비로 사용해 앱 설치, 권한 승인, 실제 BLE scan 실패 처리, 기본 화면 동작을 확인할 수 있습니다.

1

Samsung 태블릿에서 개발자 옵션 켜기

태블릿의 설정 앱에서 개발자 옵션을 활성화합니다.

  • 경로 예시: 설정 -> 태블릿 정보 -> 소프트웨어 정보 -> 빌드번호를 여러 번 누릅니다.
  • 성공 기준: 설정 화면에 개발자 옵션 메뉴가 새로 나타납니다.
2

USB debugging 켜고 Mac에 연결하기

개발자 옵션에서 USB debugging을 켠 뒤 USB 케이블로 MacBook과 태블릿을 연결합니다.

  • 태블릿에 USB 디버깅을 허용하시겠습니까? 창이 뜨면 허용합니다.
  • 성공 기준: Android Studio의 기기 선택 영역에 태블릿 이름이 표시됩니다.
3

Android Studio에서 태블릿 선택하기

상단 device dropdown에서 No Devices 대신 연결된 Samsung 태블릿을 선택합니다.

  • 실행 configuration은 app을 유지합니다.
  • 기기가 안 보이면 케이블, USB debugging, 태블릿의 연결 허용 창을 다시 확인합니다.
4

Run 버튼으로 앱 설치/실행하기

Android Studio의 초록색 Run 버튼을 누르면 debug APK가 빌드되고 태블릿에 설치된 뒤 자동 실행됩니다.

  • 이 과정은 파일을 수동으로 복사하는 것이 아니라 Android Studio가 ADB로 앱을 설치하는 흐름입니다.
  • 성공 기준: 태블릿 화면에 AIRS Wi-Fi Provisioning 앱 화면이 나타납니다.
5

태블릿에서 앱이 뜨면 확인할 것

노드가 없는 현재는 실패 화면까지 가는 것이 정상 결과입니다.

  • Bluetooth, Wi-Fi, 위치 권한 요청이 나오면 허용합니다.
  • AIRS 노드 검색 시작을 눌렀을 때 BLE scan 중 화면이 보이는지 확인합니다.
  • 노드가 없으면 timeout 뒤 AIRS 노드를 찾지 못했습니다류의 실패 메시지가 나오는지 확인합니다.
  • 검색 실패 화면에서는 다시 검색만 보이고, 선택된 노드가 없으므로 Wi-Fi 다시 입력은 보이지 않아야 합니다.

6. 노드 이름이 다를 때 수정 위치

현재 앱은 AIRS 노드를 찾을 때 Service UUID가 맞거나, BLE advertising 이름이 AIRS-SETUP-로 시작하면 AIRS 노드로 판단합니다. 실제 firmware의 노드 이름이 예를 들어 abc라면 아래 위치를 확인합니다.

수정/확인 위치 현재 기준 노드 이름이 abc라면
mobile/android/app/src/main/java/com/airs/android/provisioning/data/AirsBleConstants.kt NODE_NAME_PREFIX = "AIRS-SETUP-" NODE_NAME_PREFIX = "abc" 또는 firmware 이름 규칙에 맞는 prefix로 바꿉니다. 빈 문자열로 두면 거의 모든 BLE 기기가 매칭될 수 있으므로 피합니다.
mobile/android/app/src/main/java/com/airs/android/provisioning/data/AndroidBleProvisioningRepository.kt ScanResult.toAirsNode()가 startsWith(NODE_NAME_PREFIX)를 사용합니다. 대부분 수정하지 않습니다. 이름이 여러 패턴이면 이 함수의 filter 조건만 좁게 확장합니다.
mobile/android/app/src/main/java/com/airs/android/provisioning/data/MockBleProvisioningRepository.kt mock 표시 이름이 AIRS-SETUP-0001입니다. 실제 이름과 맞춰 데모하고 싶을 때만 mock displayName을 바꿉니다. 실제 BLE scan에는 영향이 없습니다.
firmware BLE advertising 설정 앱은 Service UUID 또는 이름 prefix 중 하나가 맞으면 노드로 봅니다. 가장 안정적인 방법은 firmware가 앱과 같은 PROVISIONING_SERVICE_UUID를 advertising에 포함하는 것입니다.

7. 실제 테스트 Step

1

노드를 설정 모드로 켜기

노드가 아직 Wi-Fi를 모르거나 버튼 입력으로 설정 모드에 들어가야 합니다.

  • 성공 기준: 노드 serial log에 BLE advertising 시작 로그가 보입니다.
  • 예상 이름: AIRS-SETUP-XXXX
2

Android 앱 설치 및 실행

Android Studio에서 Android 휴대폰을 선택하고 debug 앱을 실행합니다. 현재 휴대폰이 없으면 Samsung 태블릿으로 임시 대체 테스트할 수 있습니다.

  • 휴대폰에서 Bluetooth를 켭니다.
  • 앱 권한 요청이 나오면 Bluetooth 관련 권한을 허용합니다.
3

BLE scan 확인

앱에서 노드 검색을 시작하고 목록에 노드가 나타나는지 확인합니다.

  • 성공 기준: AIRS-SETUP-XXXX가 앱 목록에 표시됩니다.
  • 실패 시: 노드 advertising, Service UUID, Android 권한을 확인합니다.
4

노드 선택 후 GATT 연결

앱에서 노드를 선택하고 GATT 연결 및 service discovery가 되는지 확인합니다.

  • 성공 기준: AIRS Provisioning Service UUID가 발견됩니다.
  • 확인 대상: Device Info, Wi-Fi Config, Provision Status characteristic.
5

SSID/PW 입력 후 BLE write

앱에서 Wi-Fi 정보를 입력하고 노드에 JSON payload를 write합니다.

  • 성공 기준: 노드 serial log에 payload 수신 또는 received 상태가 보입니다.
  • 주의: 비밀번호 원문은 앱 로그나 노드 로그에 남기지 않습니다.
6

노드 Wi-Fi 연결 확인

노드가 전달받은 SSID/PW로 Wi-Fi 접속을 시도합니다.

  • 성공 기준: 앱에 wifi_connected 상태가 표시됩니다.
  • 실패 기준: 앱에 wifi_failed 상태가 표시되고 재입력이 가능합니다.
7

후속 확인: MQTT publish

Wi-Fi 연결 이후 노드 firmware가 MQTT publish를 수행하는지 별도 기능으로 확인합니다.

  • 이 단계는 Wi-Fi provisioning 성공 조건이 아니라 후속 시스템 검증입니다.
  • broker/backend/InfluxDB에서 sensor data 유입 여부를 확인합니다.

8. HW팀과 사전 점검할 것

점검 질문 확인 방법 막혔을 때 의심 지점
노드가 BLE 신호를 발송하나요? 노드 serial log 또는 nRF Connect 같은 BLE scanner 앱으로 확인 설정 모드 미진입, BLE advertising 미시작
노드 이름이 약속과 맞나요? AIRS-SETUP-XXXX 형태인지 확인 앱 filter와 firmware 이름 규칙 불일치
Service UUID가 앱 문서와 같나요? firmware의 BLE service UUID와 앱 문서 비교 UUID 오타, 대소문자보다 값 자체 불일치
Wi-Fi Config Characteristic이 write 가능한가요? GATT characteristic property 확인 WRITE property 누락, write with response 미지원
Status Characteristic이 notify 가능한가요? CCCD 설정과 notify property 확인 NOTIFY property 누락, descriptor 설정 누락
payload 형식이 JSON으로 합의됐나요? {"ssid":"...","password":"..."} 파싱 확인 펌웨어 parser 형식 불일치, escaping 처리 미흡
노드가 2.4GHz Wi-Fi에 붙을 수 있나요? 테스트 공유기 대역 확인 ESP32 계열은 보통 5GHz Wi-Fi 미지원
Wi-Fi 연결 결과를 BLE로 알려주나요? received, wifi_connecting, wifi_connected, wifi_failed notify 확인 앱은 성공/실패를 알 수 없어서 계속 대기
실패 시 재설정 모드로 돌아오나요? 잘못된 PW 입력 후 다시 advertising 되는지 확인 실패 후 BLE 끊김, 재시도 불가
